# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹서버 5단계(쿠키를 이용한 로그인) 제출
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 라우팅과 HttpResponse Builder Pattern 리팩토링

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Session ID를 직접 구현하되 `SecureRandom`을 이용한 CSPRNG 기반의 Session ID를 생성하기로 결정했다.
- 그렇게 판단한 이유: 단순한 UUID나 Random을 사용하는 것이 아니라 OWASP Session Management Cheat Sheet를 읽어보며 Session ID의 Entropy, PII 포함 여부, URL-safe 여부 등을 고려해야 한다는 점을 알게 되었기 때문이다.
- 고려했지만 선택하지 않은 대안: UUID 사용, Random 기반 생성, Base64URL 인코딩 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: Session 생성에 대한 테스트를 작성할 수 있었고, 보안적인 이유를 이해하면서 구현할 수 있었다. 앞으로 실제 웹 프레임워크의 Session 구현을 볼 때도 기준을 가지고 비교할 수 있을 것 같다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 로그인 기능 자체보다 Cookie 처리와 로그인 실패를 사용자에게 어떻게 알려줄지에 대한 구조를 결정하는 것이 어려웠다.
- 처음 추측한 원인: 로그인 성공 여부만 반환하면 될 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: HTTP Cookie 명세와 MDN 문서를 읽고, 브라우저의 Cookie 동작과 Redirect 방식을 확인했다.
- 실제 원인: Cookie는 여러 개가 존재할 수 있고, 로그인 실패를 표시하는 방식도 서버 렌더링과 JavaScript 방식 등 여러 선택지가 있었다.
- 해결 방법: 현재 과제 범위에서는 Redirect 후 JavaScript를 이용해 로그인 실패를 사용자에게 알려주는 방식을 선택했고, Cookie Parsing은 중복 Cookie를 예외 처리하도록 구현했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - 먼저 HTTP 명세와 브라우저 동작을 확인한다.
    - 기능 구현 전에 현재 과제 범위에서 어디까지 구현하는 것이 적절한지 먼저 판단한다.
    - 구현이 복잡해질 것 같으면 라우팅과 책임 분리를 먼저 고려한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 구현보다 먼저 OWASP Cheat Sheet, MDN, Java API 문서를 읽으면서 필요한 개념을 정리한 뒤 구현을 진행했다.
- 그 접근이 도움이 된 이유: Session ID 생성, ConcurrentMap, Cookie Parsing 등 단순히 동작하는 코드가 아니라 왜 그렇게 구현하는지를 이해하면서 개발할 수 있었다.
- 다음에도 반복하고 싶은 점: 공식 문서를 먼저 읽고 핵심 내용을 개인 노트로 정리한 뒤 구현하는 습관을 계속 유지하고 싶다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 라우팅을 계속 미루면서 RequestHandler의 분기가 점점 많아졌고 Response 생성도 스파게티 코드에 가까워졌다.
- 그렇게 된 원인: 기능 구현을 우선하다 보니 구조 개선을 계속 뒤로 미루게 되었다.
- 다음에는 다르게 해볼 구체적인 행동: 다음 단계를 진행하기 전에 라우팅 구조와 HttpResponse Builder Pattern을 먼저 리팩토링하여 이후 기능 추가가 쉬운 구조로 만든 뒤 개발을 진행한다.
