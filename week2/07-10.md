# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 직무 밋업과 스쿼드 세션을 진행하였다. 개발적으로는 많이 하지 못했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 7단계의 기능 구현을 시작하지 못했다.

### 직무 밋업

###### 가장 인상 깊었던 점

기술 스택보다도 **기록, 소통, 기본기**를 정말 중요하게 생각한다는 점이 가장 인상 깊었다. 단순히 구현하는 것이 아니라, 왜 그렇게 구현했는지 설명할 수 있는 개발자가 되어야겠다는 생각이 들었다.

###### 앞으로 해야 할 것

- 리팩토링할 때 **왜 이렇게 변경했는지** 반드시 기록하기
- 회고를 쓰는 것뿐 아니라 이전 회고도 다시 읽어보기
- 사용한 기술은 **왜 사용했고 대체제는 무엇인지**까지 공부하기
- 공식 문서(RFC, Java, Spring, OWASP)를 꾸준히 읽는 습관 유지하기
- AI는 구현보다 코드 리뷰, 테스트, 검증 용도로 적극 활용하기
- 진행 상황이나 고민을 혼자 끌지 말고 빠르게 공유하기

###### 기억에 남았던 말

> "주니어들은 무조건 삽질하시오."

바로 답을 찾기보다 먼저 고민하고, 시도하고, 실패한 과정을 기록하는 경험을 계속 쌓아가는 것이 좋겠다.

### 스쿼드 세션

###### 임현성님

- DTO, DAO가 무엇인지 몰랐다. 코드를 공부해야 할 것 같다.
- Optional을 통해서 적극적으로 NPE를 방지하신 부분이 인상 깊었다.
    - 이 내용(Java 모던 문법 사용)은 피드백에도 나온 부분이라서
    - 그런데 모던 문법을 레거시 코드에는 적용하지 못하는데 꼭 좋은 것일까? 하는 생각이 들기도 하지만, 몰라서 못 쓰는 것보다는 알고 안 쓰는 것이 훨씬 낫다는 생각이 들었다.
    - Modern Java in Action을 보고 공부하자.
        - Optional
        - Interface Mapping
- Monad 방식이 무엇인지 몰랐다.

###### 기승민님

- Router를 구현 시 함수가 lambda식의 꼴로 Map의 객체로 들어가는 것이 신기했다.
- 서브 패키지를 거리낌 없이 사용해서 내 코드와 다르게 나열되어 있지 않은 것이 좋은 것 같다.
- 단위 테스트, API 테스트 등을 다양하게 사용하신 부분이 좋았다.
- 구현 전에 설계서들을 만들고 AI와 협업한 내용을 항상 저장해 놓은 방법도 사용할 만한 것 같다는 생각이 들었다.

###### 3. 오늘 아쉬웠던 점

> 프롬프팅의 경우 인상적인 부분이 이번 주에는 없었다. 학습에 집중하느라 그냥 너무 단순하게 작성했던 것 같다.
