# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹서버 4단계 POST 회원가입 제출
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: redirect 처리와 registration 처리 부분을 if-else로 구현했기 때문에 추후 리팩토링 필요

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: HTTP 응답을 Status Line, Header, Body로 나누고, `HttpStatusCode`, `HttpStatusLine`, `HttpResponseHeader`, `HttpResponseBody`, `HttpResponse` 객체로 분리한 것
- 그렇게 판단한 이유: Response는 서버가 정하는 값이므로, 문자열을 직접 조립하기보다 HTTP 응답 구조에 맞게 객체를 분리하는 것이 유지보수에 좋다고 판단했다.
- 고려했지만 선택하지 않은 대안: 필요한 곳에서 직접 status code, header, body를 문자열로 이어 붙이는 방식
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: `toString()` 테스트를 raw string 기반으로 작성할 수 있었고, status code나 header 추가가 쉬워졌다. 다만 객체가 늘어나면서 클래스 이름과 책임을 더 명확히 해야겠다고 느꼈다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: POST 회원가입 구현 중 request body, redirect, MIME type, path traversal, GET 요청 처리 등 여러 HTTP 규칙을 동시에 고려해야 해서 구현 범위가 커졌다.
- 처음 추측한 원인: 단순히 POST body만 읽으면 회원가입 기능이 동작할 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: `Content-Length`, `Content-Type`, status line, redirect status code, Location header, GET body 의미 등을 문서와 로그를 통해 확인했다.
- 실제 원인: POST 회원가입은 body parsing만의 문제가 아니라 HTTP request/response 구조, status code, redirect, method validation, 보안 처리를 함께 고려해야 하는 문제였다.
- 해결 방법:
    - `Content-Length` 기준으로 request body를 읽도록 구현했다.
    - `GET` 요청에 body가 있으면 `BadRequestException`을 던지도록 처리했다.
    - POST 회원가입 성공 시 `302 Found`와 `Location` header를 이용해 `/index.html`로 redirect하도록 구현했다.
    - `GET`으로 회원가입을 시도하면 `405`를 반환하도록 처리했다.
    - path traversal 방지를 위해 static root와 요청 path를 normalize한 뒤 root 밖으로 벗어나는지 검사했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - HTTP 기능을 구현할 때는 먼저 request line, header, body, response status, response header를 나누어 생각한다.
    - `Content-Length`, `Content-Type`, `Location`처럼 동작에 직접 영향을 주는 header를 먼저 확인한다.
    - 보안 관련 path 처리는 문자열 비교보다 `Path`, `normalize`, `startsWith`를 이용해 검증한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: 구현 중 모르는 부분을 그냥 넘기지 않고, HTTP status code, lambda method reference, redirect 동작, GET body 의미 등을 찾아보며 정리했다.
- 그 접근이 도움이 된 이유: 단순히 동작만 맞추는 것이 아니라 왜 `302`를 쓰는지, 왜 `Location` header가 필요한지, 왜 `302`와 `307`이 다른지 이해하면서 구현할 수 있었다.
- 다음에도 반복하고 싶은 점: 기능 구현 중 새로 만난 개념은 코드만 작성하고 끝내지 말고, 짧게라도 문서화해서 다음 구현에 다시 사용할 수 있게 한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: redirection과 registration 처리 구조를 아직 깔끔하게 분리하지 못하고 if-else로 처리함.
- 그렇게 된 원인: 우선 기능을 통과시키는 데 집중하다 보니 handler 분리나 책임 분리까지 충분히 설계하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동: 기능을 먼저 동작시킨 뒤, request path와 method에 따라 처리 책임을 분리하는 방향으로 리팩토링한다. 특히 registration 같은 기능은 하드코딩된 if-else보다 별도의 handler나 controller 성격의 객체로 분리하는 것을 고려한다.
