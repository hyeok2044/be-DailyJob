# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹서버 6단계를 시작하기 전에 Response와 Request Handler를 중심으로 리팩토링을 진행했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 6단계 기능 구현은 아직 시작하지 못했다.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: `HttpResponse`를 Record Class로 변경하고, `HttpResponseBuilder`를 도입하여 응답 생성 과정을 Builder Pattern으로 리팩토링한 것.
- 그렇게 판단한 이유: 기존에는 `HttpResponse` 객체를 생성한 뒤 Setter를 여러 번 호출해야 해서 코드가 길고 읽기 어려웠다. Builder Pattern을 적용하면 응답 생성 의도가 더 명확하게 드러날 것이라고 판단했다.
- 고려했지만 선택하지 않은 대안: 기존 Setter 방식 유지, Header를 수정/삭제 가능한 Builder 구현
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: Response 생성 코드가 훨씬 간결해졌고, `HttpResponse`를 Immutable하게 만들 수 있었다. 또한 Header는 append만 가능하도록 설계하여 HTTP Response의 특성도 자연스럽게 반영할 수 있었다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 6단계를 구현하려고 보니 `RequestHandler`가 너무 많은 책임을 가지고 있어 기능을 추가하기 어려웠다.
- 처음 추측한 원인: 라우팅만 먼저 분리하면 해결될 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 현재 `RequestHandler`의 책임을 하나씩 정리하고 Response 생성 과정과 Routing을 각각 분리해 보았다.
- 실제 원인: Routing보다 Response 생성 구조가 먼저 개선되어야 했고, Builder Pattern을 적용한 뒤에야 전체 코드 구조가 단순해졌다.
- 해결 방법: `HttpResponseBuilder`, `HttpResponseUtils`를 도입하고 `HttpResponse`를 Record로 변경하여 Response 생성 과정을 리팩토링했다. 이후 Routing 구조도 조금씩 분리하기 시작했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - 기능 추가 전에 현재 클래스의 책임이 너무 많지는 않은지 먼저 확인한다.
    - 새로운 기능보다 반복되는 코드가 있다면 먼저 리팩토링을 고려한다.
    - 구현이 막히면 설계를 다시 살펴보는 시간을 갖는다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: Builder Pattern을 적용하기 전에 Builder Pattern이 왜 필요한지, 기존 생성자 방식과 Java Bean 방식의 단점을 먼저 공부한 뒤 구현했다.
- 그 접근이 도움이 된 이유: 단순히 패턴을 사용하는 것이 아니라 왜 사용하는지 이해한 상태에서 적용할 수 있었고, 이후 코드 설계에도 자신 있게 반영할 수 있었다.
- 다음에도 반복하고 싶은 점: 새로운 설계 패턴을 사용할 때는 먼저 사용 이유와 장단점을 이해한 뒤 코드에 적용하는 습관을 유지하고 싶다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 리팩토링 범위를 예상보다 크게 잡아 하루 동안 6단계 기능 구현을 시작하지 못했다.
- 그렇게 된 원인: 구조 개선을 진행하면서 계속 개선할 부분이 보여 범위가 자연스럽게 커졌다.
- 다음에는 다르게 해볼 구체적인 행동: 리팩토링의 목표를 먼저 정하고, 해당 범위까지만 마친 뒤 기능 구현으로 넘어가도록 시간을 관리한다.
