# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 별도 과제에서 `Request` 객체를 인터페이스로 변경하고 데이터 모델링 실습 중 실시간 배달 주문시스템의 논리·물리 ERD를 작성했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 데이터 모델링 실습에 집중 후 집중력이 떨어져 인터페이스 변경 이후의 코드 수정과 ERD 세부 검토를 마치지 못했다.

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: WAS 8번 과제를 수행하기 위해 `Request`의 공통 동작을 인터페이스로 정의하고 구체적인 구현과 분리했다.
- 그렇게 판단한 이유: 사용하는 측이 구체 클래스보다 필요한 동작에 의존하도록 만들기 위해서다.
- 고려했지만 선택하지 않은 대안: 기존 `Request` 클래스를 그대로 유지하는 방법을 고려했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 구현 교체와 확장이 쉬워질 기반을 만들었지만 아직 후속 리팩터링이 필요하다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 오후에 집중하지 못해 계획했던 작업을 거의 진행하지 못했다.
- 처음 추측한 원인: 피로한 상태에서 작업 범위를 구체적으로 나누지 않은 것이 원인이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 구현 범위를 줄이고 완료할 수 있는 인터페이스 변경과 ERD 작성에 집중했다.
- 실제 원인: 집중력 저하와 함께 다음 작업 순서가 명확하지 않았다.
- 해결 방법: 작업량을 줄이고 책을 읽고 이번 과제와 관련 된 지식을 쌓았다. 서로 다른 과제에서 완료 가능한 작업을 하나씩 끝냈다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 작업을 시작하기 전에 한 시간 이내에 끝낼 수 있는 단위로 나눈다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 집중이 잘되지 않는 상황에서도 인터페이스 변경과 ERD 작성을 완료했다.
- 그 접근이 도움이 된 이유: 하루를 중단하지 않고 각 과제에서 최소한의 결과를 남길 수 있었다.
- 다음에도 반복하고 싶은 점: 컨디션이 좋지 않을 때는 범위를 줄여 작은 작업이라도 완료한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 집중하지 못한 상태로 시간을 보내며 작업 진척이 적었다.
- 그렇게 된 원인: 오후 작업의 우선순위와 완료 기준을 미리 정하지 않았다.
- 다음에는 다르게 해볼 구체적인 행동: 오후 작업 시작 전에 할 일을 세 개 이하로 나누고 완료 기준을 적는다.
