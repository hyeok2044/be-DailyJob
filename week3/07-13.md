# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹서버 7단계 기능 중 Multipart, 프로필 사진 수정, 글 작성, Comment 구현 초반
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 7단계의 기능 중 Comment 조회, 좋아요 구현 등

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: InputStream, Reader 등 reader과 관련된 함수 공부하고 구현해보기
- 그렇게 판단한 이유: input관련된 부분, 문제가 생기면 어떤 결과가 나오는지 확실히 알아야 제대로 된 코드를 작성할 수 있겠다고 판단했다.
- 고려했지만 선택하지 않은 대안: AI에 관련 구현 맡기기
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 코드에는 직접적인 영향은 없지만 기록해놓고 나중에 유지 보수를 진행할 때 기록을 확인해보는 등 도움이 될 것이다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 고민을 많이하다가 article에서의 default behavior, main에서의 default behavior를 구분하기 위해 end point를 분리하겠다는 생각을 하게 되었다.

  이렇게 되니 한 가지 기능을 추가할 때 Router의 두 부분 모두 다 적용시켜야했다. 이렇게 반복을 하며 한 부분에 실수로 적용하지 않은 부분들이 생기게 되었고 이 문제를 처음 겪었을 때는 계속 같은 부분에서 재시도하는 방식으로 디버깅했다.

- 해결 방법: 두 개의 코드를 각각 신경쓰며 프로그래밍 하였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 지금은 구현을 끝 마쳤지만 리팩토링을 진행할 것이다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 구현을 진행하면서 막히는 부분이 있어도 기능을 억지로 이어 붙이기보다 Router 구조를 어떻게 가져갈지 계속 고민하며 개발했다.
- 그 접근이 도움이 된 이유: 비록 현재는 Router에 중복이 생겼지만, 왜 이런 구조적인 문제가 발생하는지 직접 경험할 수 있었고 리팩토링해야 하는 이유도 명확하게 알게 되었다.
- 다음에도 반복하고 싶은 점: 구현만 끝내는 것에 만족하지 않고, 코드 구조에서 불편한 점이 보이면 메모해 두었다가 리팩토링으로 개선하는 습관을 유지하고 싶다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 7단계 구현 중 char → byte[] 배열로 바꾸기만 하면 되는 문제인줄 알고 AI에게 조금 goal-based로 요청했는데 AI가 다 해결해버려서 어려운 부분을 쉽게 넘어갔다.
- 그렇게 된 원인: 프롬프팅을 꼼꼼하게 하지 않았다.
- 다음에는 다르게 해볼 구체적인 행동: AI가 코드를 짜주면 조금 더 자세히 볼 필요가 있겠다는 생각이 든다.
