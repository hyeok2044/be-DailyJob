# be-DailyJob 

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다. 
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습: 웹서버 1단계 제출
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 없음.

###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: 아직은 배우는 과정이라 의미 있는 기술적 판단을 내렸다고 느끼지는 못했다.
- 그렇게 판단한 이유: 구현과 학습 자체에 집중한 단계였고, 설계나 기술 선택을 주도적으로 결정할 상황은 많지 않았다.
- 고려했지만 선택하지 않은 대안: 없음.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 다음 회차부터는 구현 과정에서 어떤 선택을 했고 왜 그렇게 했는지 의식적으로 기록해 볼 것이다.

###### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: path가 맞는 것 같음에도 불구하고 경로를 찾지 못했고, 여러 로그를 찍어도 결과가 바뀌지 않았다.
- 처음 추측한 원인: Gradle 빌드 덮어쓰기 실패
- 원인을 확인하기 위해 시도한 방법: `gradle clean` 실행 및 IntelliJ 재시작
- 실제 원인: `try` 문에서 예외가 발생해 `catch` 문으로 이동하면서 이후의 디버그 로그가 실행되지 않았다.
- 해결 방법: 생성형 AI와의 질의응답을 통해 예외 발생 지점을 확인했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - 오류가 발생하는 위치를 먼저 좁혀 나가는 체계적인 디버깅을 한다.
    - `catch` 문에서도 반드시 로그를 남겨 예외를 확인한다.
    - `path`와 같은 상대 경로는 먼저 실제 값을 출력해 확인한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지: JUnit, Concurrent처럼 모르는 개념은 생성형 AI로 빠르게 학습하고, 구현은 직접 작성하려고 했다.
- 그 접근이 도움이 된 이유: AI에 과도하게 의존하지 않으면서도 필요한 학습은 빠르게 진행할 수 있었다.
- 다음에도 반복하고 싶은 점: 모르는 것은 빠르게 질문하되, 답을 그대로 사용하기보다 이해하고 직접 구현하는 방식을 유지한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 문제를 꼼꼼히 읽지 않아 1번 Task를 놓칠 뻔했다.
- 그렇게 된 원인: 꼼꼼하지 못한 습관과 다소 늦은 시간에 과제를 진행한 영향이 있었다.
- 다음에는 다르게 해볼 구체적인 행동: 구현을 시작하기 전에 요구사항을 한 번 정리하고, 생성형 AI의 요약과 비교하여 누락된 내용이 없는지 교차 검증한 뒤 개발을 시작한다.
