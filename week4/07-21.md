> 각 항목은 길게 쓰기보다, 사실·판단·근거·다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: Spring API 강의를 시청하고, 프로젝트의 패키지 구조·API 버저닝·공통 응답 형식·인증 방식·Swagger 설정에 대한 팀 논의를 진행했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: ERD와 웹 화면 프로토타입을 완성하지 못했고, 배송 상태별 입력·출력과 API 명세도 구체화하지 못했다.

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 실시간 통신은 전체 배송 과정이 아니라 초기 매칭 과정에 우선 적용하고, 배송 중 위치 갱신은 일정 주기의 API 요청으로 처리하는 방향을 검토했다.
- 그렇게 판단한 이유: 배송 전 과정에 WebSocket을 적용하면 연결 복구, 브라우저 종료, heartbeat, 재접속 처리까지 구현 범위가 크게 증가하기 때문이다.
- 고려했지만 선택하지 않은 대안: 매칭부터 배송 완료까지 WebSocket 연결을 계속 유지하며 위치와 상태를 실시간으로 동기화하는 방식.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 실시간 통신의 책임 범위를 줄여 연결 상태에 따른 예외를 단순화하고, 매칭과 배송 위치 갱신을 독립적으로 테스트할 수 있게 되었다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 배송 중 위치 추적과 연결 종료 상황을 어디까지 MVP에 포함해야 하는지 범위를 정하기 어려웠다.
- 처음 추측한 원인: 실제 서비스 수준의 실시간 위치 추적과 예외 대응을 모두 구현해야 한다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 시연 환경, GPS 정확도, 모바일 웹의 백그라운드 동작, WebSocket 재연결과 알림 수단을 팀원들과 비교했다.
- 실제 원인: 구현 범위보다 실제 운영 상황과 예외 케이스를 먼저 확장해서 생각해 MVP의 우선순위가 흐려진 것이 문제였다.
- 해결 방법: 시연에서 반드시 보여줘야 하는 매칭·픽업·배송 상태 전이를 우선 구현하고, 위치는 가짜 데이터 또는 주기적 API 갱신으로 대체하는 방향으로 범위를 줄였다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 해당 기능이 핵심 사용자 흐름과 시연에 반드시 필요한지, 더 단순한 대체 구현이 가능한지 먼저 확인한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 패키지 구조, 생성자 주입, API 버저닝, 공통 응답 형식 등 구현 전에 필요한 팀 개발 규칙을 함께 논의했다.
- 그 접근이 도움이 된 이유: 각자 다른 방식으로 코드를 작성하면서 생길 수 있는 구조적 불일치와 반복적인 수정 비용을 사전에 줄일 수 있었다.
- 다음에도 반복하고 싶은 점: 기능 구현 전에 팀 공통 규칙과 기술 선택의 근거를 명확히 정리하고, 확정된 내용은 예제 코드와 함께 문서화한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 공통 응답 형식과 WebSocket 예외 상황을 길게 논의했지만, 실제 ERD와 API 명세 작성까지 연결하지 못했다.
- 그렇게 된 원인: 기술적인 선택지를 비교하는 데 집중해 오늘 반드시 완료해야 할 산출물의 우선순위를 놓쳤다.
- 다음에는 다르게 해볼 구체적인 행동: 회의 시작 전에 완료할 산출물을 먼저 정하고, 각 논의가 끝날 때마다 ERD 테이블·상태 전이·API 요청 및 응답 명세에 즉시 반영한다.
