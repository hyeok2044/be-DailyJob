> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 배송의 정상·취소·사고·전문 기사 인계 흐름을 포함한 State Diagram 초안을 작성하고, MVP 범위·선결제·배송 인증 방식에 대한 기획 회의를 진행했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 논리 ERD 초안만 검토했으며, 엔티티 관계·제약조건을 확정한 물리 ERD 작성은 마치지 못했다.

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 배송 중 수행 불가 상황은 일반 드리미 재배정으로 복잡하게 처리하지 않고, 전문 퀵 기사 인계 또는 반송으로 종료하는 흐름으로 정리했다.
- 그렇게 판단한 이유: 물품 인수 후 재매칭은 책임 소재, 인증, 보상 처리까지 함께 복잡해져 MVP 범위를 크게 넘어서기 때문이다.
- 고려했지만 선택하지 않은 대안: 물품을 다른 드리미에게 직접 재인계하는 재배정 흐름.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 상태 전이와 예외 테스트 범위가 명확해지고, 이후 전문 기사 연동을 별도 기능으로 확장할 수 있다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: ERD를 작성하려 했지만 주문·결제·정산·배송·인증·사고 처리의 경계가 한 번에 확정되지 않았다.
- 처음 추측한 원인: 테이블 수가 많아서 구조를 단순화하기 어렵다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: State Diagram을 먼저 작성하고, 회의에서 선결제·배송 인증·취소 시점별 정책을 구체화했다.
- 실제 원인: 화면이나 테이블보다 먼저 결정되어야 할 업무 규칙과 상태 전이가 일부 남아 있었다.
- 해결 방법: 주문-결제 1:1, 배송 인증 단일 테이블, 픽업 전후 취소 구분 등 핵심 규칙을 우선 확정했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 엔티티부터 그리지 않고, 상태 전이·책임 주체·데이터가 변경되는 시점을 먼저 정리한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 단순 정상 배송 흐름뿐 아니라 지연·파손·수행 불가·취소·정산까지 예외 흐름을 함께 논의했다.
- 그 접근이 도움이 된 이유: 나중에 ERD와 API를 만들 때 빠지기 쉬운 상태, 이력, 보상 데이터를 미리 발견할 수 있었다.
- 다음에도 반복하고 싶은 점: 기능을 확정할 때 정상 플로우와 실패·취소 플로우를 항상 같이 검토한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 회의에서 나온 정책을 바로 ERD의 관계와 제약조건으로 옮기지 못해, 논의 결과가 초안 수준에 머물렀다.
- 그렇게 된 원인: State Diagram, 결제 구조, 인증 방식 등 여러 주제를 동시에 검토해 설계 산출물의 완료 기준을 정하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동: 내일은 상태별로 생성·변경되는 엔티티를 표로 정리한 뒤, 핵심 주문 흐름부터 물리 ERD와 API 명세에 순서대로 반영한다.
