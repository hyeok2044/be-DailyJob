###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 포인트와 머니의 현재 잔액은 지갑 테이블에 저장하되, 실제 자금 흐름의 원천은 원장 테이블로 관리하고 원장 변경 결과를 기준으로 지갑 잔액을 갱신하기로 했다.
- 그렇게 판단한 이유: 지갑 테이블만 직접 증감하면 데이터 불일치가 발생했을 때 원인을 추적하기 어렵다. 반면 원장을 append-only 형태로 관리하면 결제·환불·정산·환전의 이력을 보존하면서 현재 잔액도 다시 계산할 수 있다.
- 고려했지만 선택하지 않은 대안: `held_balance`, `pending_earnings_balance`와 같은 값을 모두 지갑 테이블의 컬럼으로 저장하거나, 지갑 잔액만 직접 수정하고 별도의 원장을 두지 않는 방식.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 자금 이동의 원인을 추적하기 쉬워지고, 환불과 정산을 기존 거래 수정이 아닌 별도 거래로 기록할 수 있다. 다만 지갑과 원장을 하나의 트랜잭션으로 갱신하고 불일치를 검증하는 로직이 필요하다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 포인트 결제, 드리미 수익 정산, 머니의 포인트 전환을 하나의 지갑과 거래 구조로 처리할 수 있는지 혼란스러웠다.
- 처음 추측한 원인: 포인트와 머니 모두 금액이므로 하나의 계정과 하나의 거래 테이블에서 관리할 수 있다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 부르미의 결제·환불 흐름과 드리미의 정산·출금·환전 흐름을 주문 상태에 맞춰 각각 정리하고, Wallet·Transaction·Ledger의 역할을 구분했다.
- 실제 원인: 포인트와 머니는 사용 목적과 생성 시점이 다르며, 머니에는 `PENDING → SETTLED`라는 정산 과정이 존재한다. 두 자금을 하나의 라이프사이클로 보려 한 것이 문제였다.
- 해결 방법: 포인트와 머니의 거래·원장을 분리하고, 머니는 배송 완료 시 `PENDING`으로 생성한 뒤 운영자 승인 시 사용 가능한 금액으로 전환하도록 정리했다. 머니에서 포인트로의 전환은 `EXCHANGE`를 기준으로 양쪽 원장에 각각 기록하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 금액이라는 공통점보다 자금의 생성 원인, 사용 목적, 사용 가능 시점, 취소 가능 여부를 먼저 구분한다.
