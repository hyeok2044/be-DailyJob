# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실·판단·근거·다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 팀원들과 ERD 물리 설계를 진행하고, 세션 기반 인증 방식·JPA와 JDBC 사용 방향·주소 저장 방식·거리 및 예상 시간 저장 구조를 논의했다. 또한 배송 요청자와 수행자 화면의 와이어프레임을 검토하고, 포인트 결제·본인 인증·드리미 심사 방식 등 시연 범위를 구체화했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 전체 ERD의 테이블과 관계를 확정하지 못했고, WebSocket 통신 방식과 배차 우선순위 알고리즘도 구체적인 구현 수준까지 정리하지 못했다.

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 현재 프로젝트는 단일 서버로 운영할 예정이므로 JWT보다 `HttpSession`을 사용하고, 동일 사용자의 중복 세션을 제한하기 위해 `UserId → SessionId` 매핑을 별도로 관리하는 방향을 선택했다.
- 그렇게 판단한 이유: 분산 서버 환경이 아니기 때문에 Tomcat의 기본 세션 관리 기능만으로도 인증 상태를 충분히 유지할 수 있고, JWT의 발급·검증·폐기 로직을 직접 구현하는 것보다 현재 MVP 범위에 적합하다고 판단했다.
- 고려했지만 선택하지 않은 대안: JWT 기반 무상태 인증을 구현하거나, 처음부터 Redis와 같은 외부 세션 저장소를 사용하는 방식.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 초기 인증 구현은 단순해졌지만, 서버 재시작 시 세션 소실과 다중 서버 확장에 제약이 생긴다. 따라서 세션 관리 로직을 별도 인증 계층으로 분리하고, 추후 외부 저장소로 교체할 수 있도록 직접 만든 사용자-세션 매핑이 비즈니스 코드에 퍼지지 않게 해야 한다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: 배송 예상 시간과 주소 정보를 어느 시점에 계산하고, 어떤 값을 데이터베이스에 저장해야 하는지 명확하지 않았다.
- 처음 추측한 원인: 현재 사용자 좌표를 기준으로 매번 카카오 길찾기 API를 호출하면 가장 정확한 예상 시간을 제공할 수 있다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 부르미와 드리미 사이의 이동 시간, 픽업 후 출발지와 도착지 사이의 배송 시간을 분리해서 검토하고, 클라이언트 주소 검색 결과와 위치 좌표를 서버에 전달하는 방식을 비교했다.
- 실제 원인: 주소 검색, 현재 위치, 예상 시간 계산을 하나의 문제로 보고 있었기 때문에 데이터의 책임과 갱신 시점을 구분하지 못한 것이 원인이었다.
- 해결 방법: 주소 검색은 클라이언트에서 처리하되 서버에는 검증 가능한 주소 정보와 위도·경도를 저장하고, 예상 시간은 `드리미 → 픽업지`와 `픽업지 → 도착지`를 구분하여 저장하는 방향으로 정리했다. 초기 프로토타입에서는 경과 시간 중심으로 구현하고, 이후 거리와 경로를 반영하도록 범위를 단계적으로 확장하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 외부 API를 호출하기 전에 어떤 데이터가 원본인지, 서버가 반드시 저장해야 하는 값은 무엇인지, 해당 값이 언제 다시 계산되어야 하는지를 먼저 구분한다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 인증 방식, PK 설계, 주소 처리, 평점 타입, 결제 방식처럼 구현 중 문제가 될 수 있는 항목을 코드 작성 전에 팀원들과 논의했다.
- 그 접근이 도움이 된 이유: 단순히 ERD 컬럼을 채우는 데 그치지 않고, 서버 재시작·중복 로그인·외부 API 오남용·결제 환불·평점 집계와 같은 실제 운영 상황까지 고려할 수 있었다.
- 다음에도 반복하고 싶은 점: 설계 항목마다 현재 MVP에서 필요한 방식과 추후 확장 시 보완할 방식을 구분해서 기록하고, 결정된 내용은 바로 ERD와 API 명세에 반영한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 인증, 결제, 지도 API, WebSocket, 리뷰, 본인 인증 등 논의 범위가 넓어지면서 가장 우선순위가 높았던 ERD 물리 설계를 완전히 확정하지 못했다.
- 그렇게 된 원인: 각 기능에서 발생할 수 있는 운영 예외를 깊게 검토했지만, 시연에 반드시 필요한 기능과 후순위 기능을 명확히 나누지 않은 채 논의를 계속했다.
- 다음에는 다르게 해볼 구체적인 행동: 회의 시작 시 반드시 확정할 테이블과 컬럼 목록을 먼저 정하고, 파손·환불·범죄경력 검증·앱 패키징처럼 MVP 이후에 다룰 내용은 별도의 백로그로 분리한다. ERD 논의가 끝난 뒤에는 각 상태 전이에 필요한 요청값, 응답값, 저장값을 API 단위로 바로 작성한다.
