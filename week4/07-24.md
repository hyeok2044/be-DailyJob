# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라, 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실·판단·근거·다음 행동이 드러나게 구체적으로 작성하세요.

###### 1. 오늘 완료한 것

- 오늘 실제로 끝낸 기능/수정/학습: 사용자 생년월일을 MySQL `DATE`와 JPA `LocalDate`로 저장하는 방향을 정하고, 운영 설정은 `application.properties`의 placeholder와 EC2의 환경변수 파일을 이용해 관리하기로 했다. JPA 연관관계에서는 `@ManyToOne`, `@OneToMany`와 지연 로딩을 기본으로 검토했으며, 본인인증으로 받은 CI와 개인정보를 별도 테이블로 분리하는 구조도 논의했다. 또한 포인트·결제 API 명세와 와이어프레임을 검토하고, 프론트엔드 배포는 S3와 CloudFront를 사용하는 방향을 정리했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분: 공통 응답 형식과 Swagger 문서화를 함께 적용하는 구체적인 방법을 확정하지 못했고, 에셋 저장 위치와 일부 포인트·결제 API 명세도 마무리하지 못했다. 스프링 기본 사용법과 JPA 연관관계의 실제 구현 역시 추가 학습이 필요하다.

###### 2. 오늘 가장 의미있었던 기술적 판단

- 내가 내린 판단: 현재 단일 EC2 기반 MVP에서는 AWS Secrets Manager를 바로 도입하지 않고, Git 저장소와 분리된 systemd 환경변수 파일을 사용해 운영 설정과 비밀값을 관리하기로 했다.
- 그렇게 판단한 이유: 관리해야 할 비밀값이 많지 않은 단일 서버 환경에서는 Secrets Manager 도입으로 배포 구조와 권한 설정이 복잡해지는 것에 비해 얻는 효용이 크지 않다고 판단했다.
- 고려했지만 선택하지 않은 대안: 처음부터 AWS Secrets Manager 또는 Parameter Store를 사용해 데이터베이스 비밀번호와 외부 API 키를 관리하는 방식.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 초기 배포 구조를 단순하게 유지할 수 있지만, 환경변수 파일의 접근 권한과 배포 과정에서의 노출 방지가 중요해졌다. 이후 서버 다중화, 비밀값 자동 회전, 접근 감사가 필요해지면 외부 비밀 관리 서비스로 교체할 수 있도록 애플리케이션은 환경변수에만 의존하게 해야 한다.

###### 3. 가장 막혔던 문제와 해결 과정

- 문제 상황 또는 증상: ERD, 프론트엔드 와이어프레임, API 명세, 배포 구조, 개인정보 처리와 같은 여러 주제를 동시에 논의하면서 팀 회의가 길어지고 구현 작업에 집중할 시간이 줄어들었다.
- 처음 추측한 원인: 결정해야 할 기술 항목이 많아 모든 내용을 팀원 전체가 함께 검토해야 한다고 생각했다.
- 원인을 확인하기 위해 시도한 방법: 스쿼드 세션에서 다른 팀의 협업 방식과 설계 과정을 확인하고, GitHub Issue·Discussion·Wiki 사용 방식과 주 담당자·부 담당자를 나누는 방법을 비교했다.
- 실제 원인: 회의의 대주제, 제한 시간, 완료 기준이 명확하지 않아 현재 주제와 직접 관련 없는 내용까지 함께 논의한 것이 문제였다.
- 해결 방법: 단체 논의가 필요한 경우 대주제를 먼저 정하고 타이머 안에서만 결정하기로 했다. 현재 주제와 관련 없는 내용은 별도 드래프트나 백로그에 기록하고, 아침·오후 2시·저녁처럼 정해진 시간에만 팀 회의를 진행하기로 했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것: 이 내용이 지금 전체 팀이 함께 결정해야 하는 문제인지, 담당자가 먼저 조사한 뒤 공유해도 되는 문제인지 구분한다. 회의 전에는 반드시 결정할 항목과 산출물을 적는다.

###### 4. 오늘 잘한 행동(개발과 관련된 것)

- 어떤 행동이었는지: 날짜 타입, 환경설정 관리, JPA 연관관계, 개인정보 분리, 포인트 충전 단위, 프론트 배포 방식 등 실제 구현 전에 문제가 될 수 있는 항목을 구체적으로 검토했다. 또한 스쿼드 세션에서 다른 팀의 실시간 경매, 배송 매칭, Redis 위치 저장, GitHub 문서화 방식을 살펴보고 우리 팀의 진행 방식과 비교했다.
- 그 접근이 도움이 된 이유: 단순히 기능을 구현하는 데 그치지 않고, CI 노출 위험, 서버 재배포 시 설정 관리, 실시간 위치 데이터 저장 범위, 협업 과정의 병목처럼 이후 문제가 될 요소를 미리 발견할 수 있었다.
- 다음에도 반복하고 싶은 점: 다른 팀의 결과물에서 기술 자체뿐 아니라 기획 범위 축소, 역할 분담, Wiki와 Issue 활용 방식까지 함께 관찰하고 우리 프로젝트에 적용할 수 있는 행동으로 정리한다.

###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

- 비효율적이었던 접근/놓치거나 부족했던 점: 논의와 자료 조사는 많이 진행했지만, 공통 응답과 Swagger 문제 해결, API 명세 작성, 스프링 구현처럼 실제 코드와 산출물로 연결된 결과는 상대적으로 적었다.
- 그렇게 된 원인: 여러 주제를 동시에 검토하면서 각 논의의 종료 조건을 정하지 않았고, AI가 생성한 내용도 충분히 이해하고 정리하기 전에 다음 주제로 넘어간 부분이 있었다.
- 다음에는 다르게 해볼 구체적인 행동: 스프링의 기본 사용법과 담당 기능을 먼저 학습하고, AI가 작성한 코드와 문서는 직접 설명할 수 있을 정도로 검토한다. 팀원에게 공유하기 전에는 내용과 근거를 정리하고, 사용한 프롬프트도 함께 공유한다. 회의 사이의 시간에는 개인 담당 작업에 집중하고, 논의 결과는 바로 GitHub Issue나 Wiki에 반영한다.
